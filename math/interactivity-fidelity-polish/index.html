What is this talk?

I'd like to share what I've learned about making great visualizations

This is a short, fast-paced talk with a lot of advice, some points more subjective than others

My goal is to share design principles I've learned in the hopes that it helps, never to tell you what you're doing is wrong

This talk is directed to developers of web applets primarily, but it should be broadly applicable

Target audiences

Math visualizations can interface with three very different groups: mathematicians, non-mathematician enthusiasts, and students

Often only one or two of these are considered

Some different primary goals between these audiences: visual fidelity, low barrier to entry, interactivity, clarity of explanation, etc.

I believe nearly all visualizations can appeal to all three groups, and that they're at their best when they do

Fidelity

Code performance matters!

There is a visual threshold between something feeling like a rendering and actually manipulating an object (especially important when those objects are unusual)

Use the GPU whenever possible. On the web, this looks like WebGL or (eventually) WebGPU

CPU

GPU: 400x pixels at same fps on an M3 MacBook Pro

Single-threaded performance

If something can't be parallelized but is performance-critical, you may be able to write that specific part in a lower-level language

For example, WebAssembly lets compiled C code be run from JS

It's used here in the tight loop for finding random walks in Wilson's Algorithm

Performance compromises

When it's not possible to have the best of both worlds and render at a high resolution and framerate, at least get each world individually

Rendering realtime previews at a low resolution and single renders at high resolution is a good compromise

Expandability

Expect future advances in technology: don't set hard limits on resolution, framerate, etc. by today's screens

Eventually, HDR will be one of these items (WebGPU supports it now)

If possible, let (advanced) users write small amounts of code to customize behavior

Interactivity

Draw things where they are! Prioritize controls that live in the space

When that's not possible, realtime and direct controls like sliders are better than indirect ones like text fields

Metric: 2.0

Clarity

This is a broad topic, but one thing that can help is (sparing) use of animation

Showing things moving or being drawn can help the user keep their place

Animation as education

Showing objects animating can also have educational value

This is the beta reduction process for a lambda calculus Tromp diagram; the full applet allows the animation to be slowed down or paused

Animation as substance

Sometimes the animation is an intrinsic part of the value

The growth of a Gravner–Griffeath snowflake is as much about the process as the final result

Animation as flair

There is still a place for animation outside of these categories, but if you're only animating between two successive user inputs, be conservative

If you're going to keep animations for beauty, make sure they're merited, reasonably short, and/or skippable

Accessibility

A high skill ceiling should not preclude a low skill floor

It should be easy to directly manipulate the scene in a way that aligns with users' intuitions

For 2D scenes, that usually looks like Google-Maps-style panning and zooming with inertia

3D scenes

For 3D objects, a natural interaction method is rotating

Take caution when using video-game-like controls for looking around as the default, since users can accidentally lose sight of the entire scene

Minimize barrier to entry

Provide sensible and pleasant defaults to parameters and quality, and if your applet requires writing code, add nice examples that don't require copy-pasting

Most importantly, build with (small) touchscreens in mind! There are always edge cases that degrade user experience if unconsidered

Beauty

By far the most subjective topic in this talk!

When choosing color, try to match the color space to what you're building: for example, RGB works well for colors from $xyz$-coordinates, and HSV is a natural choice when one quantity is circular (e.g. an angle)

Color palettes

When coloring something more discrete, you can generate a palette randomly, but it can be best to restrict the color possibilities to produce something pleasant

For example, random hues without intense greens, medium-low saturations, high values

Adding sound

Taking an existing process or scene and adding sound can provide a novel and intuitive path to understanding it

Most famously done with sorting algorithms

Designing around sound

It's also possible to design an applet from the ground up to play sound, typically by constructing a sound wave out of individual samples

YouTuber CodeParade did this with orbits in Mandelbrot sets, and 2Swap with state space paths of double pendulums
